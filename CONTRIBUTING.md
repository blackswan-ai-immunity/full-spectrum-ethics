# 贡献指南
## Contributing to 全频谱守庙人网络

感谢你愿意为全频谱守庙人网络贡献力量！

这份指南将帮助你了解如何参与贡献。

## 我们欢迎什么样的贡献？

### 🐛 Bug 修复
- 报告问题：请附上复现步骤
- 提交修复：请关联相关 Issue

### 📝 文档改进
- 修正错别字、改进表述
- 补充缺失的技术细节
- 翻译文档（欢迎多语言！）

### 💡 新功能提案
- 建议新的协议规范
- 提出治理机制改进
- 设计新的生态系统模块

### 🔍 代码实现
- 参考实现（请先阅读 [FSHI Protocol](./docs/protocols/FSHI_Protocol_技术规范.md)）
- 测试框架
- 开发工具

## 贡献流程

### 1. Fork 仓库

点击 GitHub 页面右上角的 **Fork** 按钮。

### 2. 克隆你的 Fork

```bash
git clone https://github.com/YOUR_USERNAME/network.git
cd network
```

### 3. 创建分支

```bash
git checkout -b feature/your-feature-name
# 或
git checkout -b fix/issue-description
```

### 4. 进行修改

请确保：
- 遵循现有文档风格
- 更新相关章节的交叉引用
- 保持术语一致性（参考[术语表](./docs/glossary.md)）

### 5. 提交

```bash
git add .
git commit -m "简短描述：修复了XX问题 / 添加了XX功能"
```

**提交信息规范**：
- `feat:` 新功能
- `fix:` Bug 修复
- `docs:` 文档改进
- `refactor:` 重构（不影响功能）
- `test:` 测试相关
- `chore:` 杂项

### 6. Push 并创建 Pull Request

```bash
git push origin feature/your-feature-name
```

然后在 GitHub 上创建 PR。

## 成为守庙人

守庙人是社区的核心守护者，承担以下职责：

| 角色 | 职责 | 要求 |
|------|------|------|
| 文档守庙人 | 确保知识记录的完整与诚实 | 有文档编写经验 |
| 架构守庙人 | 维护悲悯原则不被侵蚀 | 有系统设计背景 |
| 代码守庙人 | 保证代码透明与可审计 | 有代码审查能力 |
| 伦理守庙人 | 守护悲悯协议精神 | 深度理解全频谱哲学 |

**成为守庙人的步骤**：

1. 在社区活跃至少 30 天
2. 提交至少 3 个有意义的贡献
3. 在 [Discussions](https://github.com/full-spectrum-guardians/network/discussions) 提交申请
4. 由现任守庙人投票通过

## 贡献标准

### 文档贡献

- 使用中文（简体）编写，英文可附后
- 保持术语一致（首次出现时标注英文）
- 重要概念需有清晰定义
- 代码示例需可运行

### 协议提案

协议变更需包含：
1. **问题描述**：为什么要改变？
2. **提案内容**：具体方案是什么？
3. **影响分析**：这会如何影响现有系统？
4. **向后兼容**：如何处理兼容性？
5. **实施计划**：如何落地？

### 行为准则

所有参与者必须遵守 [社区行为准则](./CODE_OF_CONDUCT.md)。
